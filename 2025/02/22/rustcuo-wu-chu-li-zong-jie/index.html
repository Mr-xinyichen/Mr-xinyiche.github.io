Rust错误处理

Awsome

更新日期:   2025-02-22
文章字数:   984
阅读时长:   4 分

Rust 提供了两种主要的错误处理方式：

  1. 不可恢复错误（Unrecoverable Errors）：使用 panic! 宏。
  2. 可恢复错误（Recoverable Errors）：使用 Result<T, E> 类型。

错误处理在 Rust 中至关重要，既可以提升代码的健壮性，又能提高可读性和维护性。本章将详细介绍 Rust 的错误处理机制，并给出相应的示例。

9.1 不可恢复错误 (panic!)

当程序遇到严重错误无法继续运行时，Rust 提供 panic! 宏来立即终止程序，并打印错误消息和回溯信息。

1. panic! 的基本使用

panic! 会导致程序崩溃，并输出错误信息。

fn main() {
    panic!("发生了一个不可恢复的错误！");
}

运行该程序时，输出类似：

thread 'main' panicked at '发生了一个不可恢复的错误！', src/main.rs:2:5
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

2. 数组索引越界导致 panic!

fn main() {
    let v = vec![1, 2, 3];
    println!("{}", v[99]); // 数组索引越界，触发 panic
}

运行该程序时，输出类似：

thread 'main' panicked at src\main.rs:2:21:
index out of bounds: the len is 3 but the index is 99
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

这类 panic! 发生在运行时，主要是由于数组访问越界、空指针解引用等原因。

3. 使用 RUST_BACKTRACE 获取回溯信息

可以使用环境变量 RUST_BACKTRACE=1 来查看 panic! 的详细回溯信息。

RUST_BACKTRACE=1 cargo run

4. panic! 的典型应用场景

  • 代码进入非法状态（如数组索引超出范围）。
  • 不希望程序继续运行，如关键文件缺失、系统错误等。
  • 用于开发调试，以便尽早发现错误（如 unreachable!() ）。

9.2 可恢复错误 (Result<T, E>)

对于可以处理的错误，Rust 提供了 Result<T, E> 枚举：

enum Result<T, E> {
    Ok(T),  // 成功，返回 T
    Err(E), // 失败，返回错误信息 E
}

1. 使用 match 处理 Result

use std::fs::File;
use std::io::Error;

fn read_file() -> Result<File, Error> {
    let f = File::open("hello.txt");
    match f {
        Ok(file) => Ok(file),
        Err(e) => Err(e),
    }
}

解析：

  • File::open("hello.txt") 可能成功，也可能失败。
  • 使用 match 处理 Ok 和 Err，如果失败则返回 Err(e)。

2. 使用 ? 运算符简化错误处理

? 可以自动传播错误，使代码更简洁。

fn read_file() -> Result<File, Error> {
    let f = File::open("hello.txt")?;
    Ok(f)
}

等价于：

fn read_file() -> Result<File, Error> {
    match File::open("hello.txt") {
        Ok(file) => Ok(file),
        Err(e) => return Err(e),
    }
}

3. Result 处理 read_to_string 示例

use std::fs::File;
use std::io::{self, Read};

fn read_username_from_file() -> Result<String, io::Error> {
    let mut f = File::open("hello.txt")?;
    let mut s = String::new();
    f.read_to_string(&mut s)?;
    Ok(s)
}
  • ? 使代码更简洁，自动传播错误。
  • 如果 File::open 或 read_to_string 失败，? 会将 Err(e) 直接返回。

9.3 选择 panic! 还是 Result？

1. 什么时候使用 panic!

  • 代码进入非法状态（如索引越界）。
  • 程序无法继续运行，如关键数据丢失。
  • 开发过程中暴露 bug（如 unreachable!() ）。

2. 什么时候使用 Result<T, E>

  • 错误是可恢复的，如文件未找到、网络请求失败。
  • 希望调用者决定如何处理错误。
  • 构建健壮的 API，避免 panic! 影响整个程序。

3. expect() 提供更明确的错误信息

use std::fs::File;

fn main() {
    let f = File::open("hello.txt").expect("无法打开文件 hello.txt");
}

expect() 与 unwrap() 类似，但 expect() 允许提供更具体的错误信息。

4. 综述

  • panic! 适用于 严重错误，代码无法继续执行。
  • Result 适用于 可恢复错误，让调用者决定处理方式。
  • ? 语法简化了 Result 传播，使代码更简洁可读。
